Routing Tree v0.6

2026 Architecture Draft Navigation

An early deep-nested routing plan that was later replaced by a flatter section-first structure.

Context

This version attempted to model every content branch through strict nested routes. It made logical sense on paper but introduced fragile URL depth, uneven naming, and hard-to-maintain navigation states.

1. Initial Intention

The goal was to mirror conceptual hierarchy directly in URL depth. Every sub-layer had a path segment, and each segment carried an explicit taxonomy role.

2. Why It Broke

As categories evolved, links became brittle. Renaming one middle segment propagated changes across many pages and increased maintenance friction during content iteration.

3. What Was Kept

Although discarded, this path clarified one useful principle: top navigation should expose only section-level stability, while deeper taxonomy remains internal to each section.

Archive Note

This artifact is preserved as a decision trace for future routing refactors.
